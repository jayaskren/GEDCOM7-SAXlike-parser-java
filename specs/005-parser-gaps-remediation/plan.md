# Implementation Plan: Parser Gaps Remediation

**Branch**: `005-parser-gaps-remediation` | **Date**: 2026-03-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/005-parser-gaps-remediation/spec.md`

## Summary

Remediate 10 gaps identified by evaluation agents in the GEDCOM 7 SAX-like
parser. The work spans structure validation completeness, minimum cardinality
enforcement, typed date values, public strategy interfaces, startRecord payload
delivery, xref validation, byte offset tracking, tokenizer I/O performance,
level-0 tag validation, and a SimpleGedcomHandler convenience class.

All changes are backward compatible. No new runtime dependencies.

## Technical Context

**Language/Version**: Java 11+
**Primary Dependencies**: None at runtime (zero-dependency library)
**Storage**: N/A (streaming parser, no persistence)
**Testing**: JUnit 5.11.4
**Target Platform**: JVM (any platform with Java 11+)
**Project Type**: Library
**Performance Goals**: 1.5x tokenizer throughput on 100K+ line files, avg of 5 runs after warm-up (US8)
**Constraints**: Zero runtime dependencies, backward compatible API changes
**Scale/Scope**: 10 user stories, ~16 functional requirements

## Architecture Decisions

### AD-1: Strategy Interface Visibility

**Decision**: Make `GedcomInputDecoder`, `PayloadAssembler`, and
`AtEscapeStrategy` interfaces `public` within the `internal` package.
Update `module-info.java` to export `org.gedcom7.parser.internal` as a
qualified export or move the interfaces to a new `org.gedcom7.parser.spi`
package.

**Preferred approach**: Move the three interfaces to a new
`org.gedcom7.parser.spi` package and export it in `module-info.java`.
Keep implementations in `internal`. This provides clean separation between
the SPI (Service Provider Interface) contracts and internal implementations.

**Alternative considered**: Simply making `internal` package exported.
Rejected because it would expose implementation details (GedcomLine,
tokenizer internals) that should remain hidden.

### AD-2: startRecord Value Delivery

**Decision**: Add a new default method to `GedcomHandler`:
```java
public void startRecord(int level, String xref, String tag, String value) {
    startRecord(level, xref, tag); // backward compatible default
}
```

The parser calls the 4-parameter version. Existing handlers that only
override the 3-parameter version continue to work via the default delegation.

### AD-3: GedcomDateValue Interface

**Decision**: Create `GedcomDateValue` interface in `datatype` package.
Existing classes (`GedcomDate`, `GedcomDateRange`, `GedcomDatePeriod`)
implement it. The interface provides:
- `DateValueType getType()` (enum: EXACT, RANGE, PERIOD, APPROXIMATE, UNPARSEABLE)
- `String getOriginalText()`

**Date type mapping**:
- `GedcomDate` → `EXACT` (e.g., `6 APR 1952`)
- `GedcomDateRange` with BET...AND → `RANGE`
- `GedcomDateRange` with ABT/CAL/EST → `APPROXIMATE`
- `GedcomDateRange` with BEF/AFT → `RANGE` (open-ended ranges are still
  ranges per the GEDCOM 7 spec, which groups BEF/AFT/BET under `DateRange`)
- `GedcomDatePeriod` → `PERIOD`
- Unparseable input → `UNPARSEABLE` (returned as a minimal implementation,
  not null or exception)

`parseDateValue()` return type changes from `Object` to `GedcomDateValue`.
This is source-compatible for callers using `Object` (widening still works)
and provides type safety for new callers.

### AD-4: Byte Offset Tracking

**Decision**: Track byte offsets in `GedcomLineTokenizer` by counting
bytes consumed during reading. When constructed from an `InputStream`,
the decoder wraps it in a counting stream before creating the `Reader`.
When constructed from a `Reader` directly, byte offset returns -1.

### AD-5: Buffered Tokenizer I/O

**Decision**: Replace character-by-character `Reader.read()` calls with
bulk `Reader.read(char[], off, len)` into a reusable buffer. Parse from
the buffer using index tracking. Refill buffer when exhausted. This
eliminates per-character method call overhead while preserving identical
parse behavior.

### AD-6: Minimum Cardinality Enforcement

**Decision**: At endStructure/endRecord time, check the parent's
`childCounts` map against the cardinality data from `StructureDefinitions`.
For each required child (cardinality starting with `{1:`), if the count
is 0, emit a validation warning. This reuses the existing `childCounts`
tracking already in place for max-cardinality checks.

### AD-7: SimpleGedcomHandler

**Decision**: Create a concrete class extending `GedcomHandler` in the
`org.gedcom7.parser` package. It overrides `startRecord`, `endRecord`,
`startStructure`, `endStructure` to delegate to unified callbacks:
- `onStructure(int level, String xref, String tag, String value)`
- `onEndStructure(int level, String tag)`

Document/error callbacks pass through unchanged.

## Complexity Tracking

| Deviation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Min-cardinality validation in parser (Principle VI) | Completes existing max-cardinality validation; opt-in via config | Deferring to consumers creates asymmetric validation that erodes trust per evaluation findings |
| `char[]` buffer instead of `byte[]` (Principle III) | InputDecoder already operates at byte-to-char boundary | Buffering at `byte[]` would duplicate the decoding layer and break the existing strategy pattern |
